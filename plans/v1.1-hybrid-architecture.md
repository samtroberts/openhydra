# v1.1 Hybrid Local/Swarm Architecture Blueprint

**Author**: Claude (Staff System Architect)
**Date**: 2026-04-01
**Status**: PLAN ONLY — no code changes
**Goal**: OpenHydra runs as a best-in-class local model runner (Ollama parity) when offline, and seamlessly reconnects to the global swarm on toggle.

---

## Executive Summary

OpenHydra today assumes the network is always available. Every inference call routes through `DiscoveryService` (DHT lookup) and `InferenceChain` (gRPC pipeline). On an 8 GB M1 MacBook Air with no internet, the node crashes within seconds of receiving a request because discovery fails, chain construction returns no peers, and the coordinator returns 503.

This blueprint transforms OpenHydra into a **dual-mode engine**:

- **Local Mode**: Zero network calls. MLX/PyTorch runs the full model directly. API is 100% Ollama + OpenAI compatible. Startup in < 3 seconds.
- **Swarm Mode**: Full DHT, gRPC pipeline, verification, economy. Exactly how it works today.
- **The Toggle**: A single IPC message from the Tauri UI that hot-switches between modes without restarting the Python process.

```
                   Local Mode                    Swarm Mode
              ┌───────────────────┐         ┌───────────────────┐
  curl/SDK    │  HTTP API :8080   │         │  HTTP API :8080   │
      │       │  ┌─────────────┐  │         │  ┌─────────────┐  │
      └──────►│  │ LocalEngine │  │◄──────► │  │ Coordinator │  │
              │  │  (no gRPC)  │  │ toggle  │  │  Engine     │  │
              │  └──────┬──────┘  │         │  └──────┬──────┘  │
              │         │ direct  │         │         │ gRPC    │
              │  ┌──────▼──────┐  │         │  ┌──────▼──────┐  │
```

## Pillar 1: Offline Engine Detachment

### Problem

The current inference path is: `API → InferenceService.infer() → _prepare_inference() → DiscoveryService._discover_for_model() → DHT HTTP lookup → PipelineService._select_pipeline() → InferenceChain.run() → gRPC Forward()`. Every single step after `_prepare_inference()` requires network access.

### Solution: `LocalInferenceEngine`

Create a new lightweight engine class that bypasses the entire coordinator service stack and calls `ModelShard.forward()` directly.

**File**: `coordinator/local_engine.py` (new, ~300 lines)

```
LocalInferenceEngine
├── __init__(model_id, shard: ModelShard, tokenizer)
│   - No EngineConfig, no services, no DHT, no gRPC
│   - Holds a direct reference to the ModelShard
│   - Loads tokenizer once at init (from HF cache, offline-safe)
│
├── infer(prompt, max_tokens, temperature, top_p, stop) → dict
│   - Tokenize prompt locally
│   - Call shard.forward(prompt, activation=[], max_tokens=max_tokens)
│   - Decode tokens back to text locally
│   - Return OpenAI-compatible response dict
│   - ZERO network calls
│
├── infer_stream(prompt, max_tokens, ...) → Generator[dict]
│   - Same as infer() but yields token-by-token
│   - Uses MLXRuntime's stream_generate() or PyTorchRuntime's decode loop
│   - Each yield is one SSE chunk
│
├── chat(messages, max_tokens, ...) → dict
│   - Apply chat template (Jinja2 from tokenizer config, cached locally)
│   - Delegate to infer()
│
├── chat_stream(messages, ...) → Generator[dict]
│   - Apply chat template then delegate to infer_stream()
│
└── list_models() → list[dict]
    - Return the single loaded model's metadata
    - No DHT scan, no network
```

### Key Design Decisions

1. **No gRPC at all in local mode.** The shard is called as a Python object, not over the network. This eliminates the ~5ms gRPC loopback overhead per token AND the gRPC thread pool.

2. **No coordinator services.** No `DiscoveryService`, no `PipelineService`, no `KvAffinityService`, no `HealthService`, no `EconomyService`. Local mode doesn't discover peers, doesn't build pipelines, doesn't track health, doesn't earn tokens.

3. **Tokenizer loaded once, cached.** The current flow loads the tokenizer from HF Hub on every inference call (6 HEAD requests per token!). `LocalInferenceEngine` loads it once at init. If cached in `~/.cache/huggingface/`, this is fully offline.

4. **No verification.** Mystery shopper, redundant exec, and auditor are swarm features. Local mode trusts the local shard.

### Network Calls Eliminated

| Call | Where | Frequency | Eliminated? |
|------|-------|-----------|------------|
| DHT peer lookup | `DiscoveryService._load_candidate_peers()` | Per inference | Yes |
| HF Hub HEAD checks | `InferenceService.infer()` via tokenizer | 6 per token | Yes (cached once) |
| gRPC Forward | `InferenceChain._request_stage()` | Per pipeline stage | Yes (direct call) |
| DHT announce | `_announce_loop()` | Every 60s | Yes (thread not started) |
| Health ping | `DiscoveryService._scan_network()` | Pre-inference | Yes |
| Grounding search | `GroundingClient.search()` | Per inference | Yes |

### Implementation: Unified API Handler

**File**: Modify `coordinator/api_server.py`

The API handler should detect which engine is active and delegate accordingly:

```python
class OpenHydraHandler(BaseHTTPRequestHandler):
    engine: CoordinatorEngine | None = None       # Swarm mode
    local_engine: LocalInferenceEngine | None = None  # Local mode

    @property
    def active_engine(self):
        """Return whichever engine is currently active."""
        return self.local_engine if self.local_engine else self.engine
```

For `/v1/chat/completions`, the response format is identical regardless of engine — the caller never knows whether they hit local or swarm. This is the key to the "Trojan Horse": users point Cursor at `localhost:8080` and it just works, whether the swarm is up or not.

### Token Counting Fix

Replace the word-split estimator (`len(text.split())`) with actual tokenizer counts:

```python
# In LocalInferenceEngine (and backport to CoordinatorEngine):
prompt_tokens = len(self._tokenizer.encode(prompt))
completion_tokens = len(generated_token_ids)  # From MLX/PyTorch decode
```

This requires the tokenizer to be available in the response path. In local mode it's always available. In swarm mode, cache the tokenizer on first use per model.

## Pillar 3: State Transitions (The Toggle)

### Current Process Architecture

```
openhydra-node process
├── Main Thread: HTTP API (ThreadingHTTPServer.serve_forever())
├── Daemon Thread: gRPC peer server
│   ├── gRPC ThreadPoolExecutor (16 workers)
│   ├── Announce loop (DHT HTTP every 60s)
│   └── Daemon budget loop (resource monitor)
```

The Tauri desktop app spawns this as a sidecar via `Command.sidecar("binaries/openhydra-node", args)`. Today there is NO way to change the mode without killing and restarting the process.

### Solution: Mode Switch via HTTP Control Plane

Add a new internal endpoint that the Tauri Rust backend calls:

```
POST /v1/internal/mode
Body: {"mode": "local"} or {"mode": "swarm"}
Auth: Must come from 127.0.0.1 (reject external)
```

**What happens on `{"mode": "local"}`:**

```
Step 1: API handler receives POST /v1/internal/mode {"mode": "local"}
Step 2: Signal announce thread to stop
         → Set _announce_stop_event (threading.Event)
         → Announce thread exits its sleep loop on next iteration (< 5s)
Step 3: Create LocalInferenceEngine
         → Reuse the EXISTING ModelShard from PeerService (zero reload)
         → Load tokenizer (one-time, from cache)
Step 4: Swap active engine
         → Set OpenHydraHandler.local_engine = local_engine
         → Set OpenHydraHandler.engine = None
Step 5: Return {"status": "ok", "mode": "local"}
```

**What happens on `{"mode": "swarm"}`:**

```
Step 1: API handler receives POST /v1/internal/mode {"mode": "swarm"}
Step 2: Unload local engine
         → Set OpenHydraHandler.local_engine = None
Step 3: Recreate CoordinatorEngine with DHT URLs
         → Or if we kept it alive, just re-enable it
         → Set OpenHydraHandler.engine = coordinator_engine
Step 4: Restart announce thread
         → Clear _announce_stop_event
         → Start new announce thread targeting DHT bootstrap nodes
Step 5: Wait for first successful DHT announcement (with 5s timeout)
Step 6: Return {"status": "ok", "mode": "swarm"}
```

**Critical constraint**: The ModelShard must NOT be unloaded during the switch. Both engines share the same shard. The shard holds ~1.6 GB of MLX weights in memory — reloading it would take 2+ seconds and spike memory.

### New Rust Command

```rust
#[tauri::command]
async fn switch_mode(mode: String) -> Result<String, String> {
    let client = reqwest::Client::new();
    let resp = client
        .post("http://127.0.0.1:8080/v1/internal/mode")
        .json(&serde_json::json!({"mode": mode}))
        .timeout(std::time::Duration::from_secs(10))
        .send()
        .await
        .map_err(|e| e.to_string())?;
    resp.text().await.map_err(|e| e.to_string())
}
```

## Pillar 4: VRAM / Resource Reallocation

### The Problem

In Local Mode, the user wants to run the **largest model their hardware supports** (e.g., Qwen 3.5 4B on a 16 GB Mac = 9 GB weights). In Swarm Mode, they contribute a **smaller model** (e.g., Qwen 3.5 0.8B = 1.6 GB) to leave room for the inference pipeline overhead (gRPC, KV cache relay, verification buffers).

Switching from a 4B local model to a 0.8B swarm model requires unloading 9 GB and loading 1.6 GB — a ~5 second operation on MLX.

### Solution: Staged Model Swap

**Local → Swarm transition:**

```
Step 1: Freeze inference (reject new requests with 503 + Retry-After: 5)
Step 2: Drain in-flight requests (wait up to 10s for completion)
Step 3: Unload local model
         → ModelShard._runtime = None (Python GC + MLX buffer release)
         → Force GC: gc.collect(); mx.metal.clear_cache() (if MLX)
Step 4: Load swarm model
         → Create new ToyShardConfig with swarm model ID
         → Create new ModelShard (triggers model download/cache load)
Step 5: Update PeerService.shard reference
Step 6: Resume inference (clear 503 gate)
Step 7: Start announce thread (peer now visible to swarm)
```

**Swarm → Local transition:**

```
Step 1: Stop announce thread (peer disappears from DHT in < 5 minutes)
Step 2: Freeze inference
Step 3: Drain in-flight gRPC requests
Step 4: Unload swarm model
Step 5: Load local model (the big one)
         → Use hardware recommendation: recommendModel(systemRam)
         → Or user's explicit choice from the model selector
Step 6: Create LocalInferenceEngine with new shard
Step 7: Resume inference
```

### Memory Budget Calculation

```python
def recommend_models(ram_gb: int) -> tuple[str, str]:
    """Return (local_model, swarm_model) for given RAM."""
    # Swarm model: always the smallest (0.8B = 1.6 GB)
    swarm = "Qwen/Qwen3.5-0.8B"

    # Local model: biggest that fits with 2 GB headroom
    available = ram_gb - 2  # OS + app overhead
    if available >= 16:
        local = "Qwen/Qwen3.5-9B"     # 18 GB, int8
    elif available >= 7:
        local = "Qwen/Qwen3.5-4B"     # 9 GB, int4
    elif available >= 3:
        local = "Qwen/Qwen3.5-2B"     # 5 GB, fp32
    else:
        local = "Qwen/Qwen3.5-0.8B"   # 2 GB, fp32

    return local, swarm
```

On the user's 8 GB Mac: local = Qwen 3.5 2B (5 GB), swarm = Qwen 3.5 0.8B (1.6 GB).

### MLX Memory Release

MLX uses Apple's unified memory. `mx.metal.clear_cache()` releases the Metal buffer pool but does NOT free the Python-side weight tensors. To fully release:

```python
# In ModelShard.unload():
del self._runtime._model      # Release model reference
del self._runtime._tokenizer   # Release tokenizer reference
del self._runtime              # Release runtime
self._runtime = None
import gc; gc.collect()        # Force Python GC
try:
    import mlx.core as mx
    mx.metal.clear_cache()     # Release Metal buffer pool
except ImportError:
    pass
```

## The Trojan Horse Strategy

The genius of this architecture is the **funnel**:

1. **Day 1**: User installs OpenHydra because it's a great local LLM runner (Ollama alternative with a native Mac app).
2. **Day 3**: User notices "Swarm Mode" toggle. Curiosity.
3. **Day 5**: User enables Swarm Mode. Sees HYDRA balance tick up. Hooked.
4. **Day 10**: User leaves Mac running 24/7 as a swarm node. Network grows.

The local mode must be **genuinely excellent** — not a demo, not a teaser. It must be the best way to run Qwen/Llama/Mistral on a Mac. Period. The swarm is the upsell, not the product.
